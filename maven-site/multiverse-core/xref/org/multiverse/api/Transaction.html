3   /***
4    * All changes on AtomicObjects must be done through a Transaction. The transaction make sure that the changes
5    * are:
6    * <ol>
7    * <li>Atomic: all or nothing gets committed (Failure atomicity)</li>
8    * <li>Consistent : </li>
9    * <li>Isolated: a transaction is executed isolated from other transactions. Meaning that a transaction won't
10   * see changed made by transactions executed concurrently, but it will see changes made by transaction completed
11   * before.</li>
12   * </ol>
13   * <p/>
14   * A Transaction is not thread-safe to use (just like a Hibernate Session is not thread-safe to use). It can
15   * be handed over from transaction to transaction, but one needs to be really careful with threadlocals.
16   * Although the Stm/Transaction implementation don't care about threadlocals, the stuff in front (templates,
17   * instrumentation etc) could care about threadlocals.
18   *
19   * @author Peter Veentjer.
20   */

25       * a family name. This information can be used for debugging/logging purposes but also other
26       * techniques that rely to know something about similar types of transactions like profiling.
27       *
28       * @return the familyName. The returned value can be null.
29       */
30      String getFamilyName();

35       * see a stable view of the objects at some point in time). The returned version will
36       * always be larger than Long.MIN_VALUE.

38       * @return the version of the stm when this Transaction started.

53       *
54       * @return the new version if the commit was a success. If the there are no changes, the readVersion
55       *         if the transaction is returned. Otherwise the writeVersion of the transaction is returned.
56       * @throws org.multiverse.api.exceptions.WriteConflictException

58       * @throws org.multiverse.api.exceptions.DeadTransactionException
59       *          if this transaction already is aborted.

61      long commit();

64       * Aborts this Transaction. This means that the changes made in this transaction are not
65       * committed. It depends on the implementation if this operation is simple (ditching objects
66       * for example), or if changes need to be rolled back.
67       * <p/>
68       * If the Transaction already is aborted, the call is ignored.
69       * <p/>
70       * It is important that the abort never fails.
71       *
72       * @throws org.multiverse.api.exceptions.DeadTransactionException
73       *          if this transaction already is committed
74       */
75      void abort();

77      /***

79       * committed or aborted.
80       * <p/>
81       * It could be that the same instance is returned just by resetting the instance.
82       *
83       * @throws org.multiverse.api.exceptions.ResetFailureException
84       *          if the restart has failed (for example
85       *          because the transaction is still active).

87      Transaction restart();

138     /***
139      * Registers a task to be executed <b>after</b> the transaction completes. If the transaction
140      * aborts, all tasks are discarded.
141      * <p/>
142      * If the execution of one of the tasks fails, the others won't be executed.
143      * <p/>
144      * There is no guaranteed order of execution of the tasks.
145      * <p/>
146      * If the same tasks is added multiple times, it could be executed multiple
147      * times (no guarantee).
148      * <p/>
149      * The tasks will be executed on the current thread that calls the commit.
150      * <p/>
151      * If the task accesses the STM, it could see changes made after the commit
152      * of the current transaction. So they will not be running on the same transaction.
153      * <p/>
154      * A good use case of this feature is starting up threads. If you need to
155      * start threads, you don't want to start them immediately because eventually
156      * the transaction could be aborted. And another problem is that new transaction started
157      * by spawned threads are not able to see the changes already made in the current transaction,
158      * because the current transaction hasn't completed yet.
159      *
160      * @param task the task to execute after the transaction completes.
161      * @throws NullPointerException if task is null.
162      * @throws org.multiverse.api.exceptions.DeadTransactionException
163      *                              if transaction already is aborted or committed.
164      */
165     void deferredExecute(Runnable task);
166 
167     /***
168      * Registers a task to be executed <b>after</b> the transaction aborts. If the transaction
169      * commits, all tasks are discarded.
170      * <p/>
171      * If the execution of one of the tasks fails, the others won't be executed.
172      * <p/>
173      * There is no guaranteed order of execution of the tasks.
174      * <p/>
175      * If the same tasks is added multiple times, it could be executed multiple
176      * times (no guarantee).
177      * <p/>
178      * The tasks will be executed on the current thread that calls aborts the transaction.
179      * <p/>
180      * A good use case of this feature is cleaning up resources.
181      * <p/>
182      * It is important to remember that the task is not going to see the changes
183      * made by the transaction, because the transaction aborts before the task
184      * is executed.
185      *
186      * @param task the task to execute after the transaction aborts.
187      * @throws NullPointerException if task is null.
188      * @throws org.multiverse.api.exceptions.DeadTransactionException
189      *                              if transaction already is aborted or committed.
190      */
191     void compensatingExecute(Runnable task);
